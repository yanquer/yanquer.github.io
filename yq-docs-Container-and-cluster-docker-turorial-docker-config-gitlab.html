docker配置gitlab

修改于: 2024-06-03
相关资源:
  • gitlab-ce官方docker地址: https://hub.docker.com/r/gitlab/gitlab-ce
  • docker-mac安装: https://docs.docker.com/desktop/install/mac-install/#system-requirements

安装

我用的是16版本

docker pull gitlab/gitlab-ce:16.5.8-ce.0

配置

使用 docker-compose 来执行,

创建运行环境

mkdir ~/Project/Docker/gitlab
cd ~/Project/Docker/gitlab
touch docker-compose.yml

内容

# docker-compose.yml
services:
  gitlab:
    container_name: 'gitlabCe16'    #容器名
    image: 'gitlab/gitlab-ce:16.5.8-ce.0'
    restart: always
    # hostname: 'gitlab.example.com'
    environment:
      TZ: 'Asia/Shanghai'
      GITLAB_OMNIBUS_CONFIG: |
        external_url  'http://192.168.1.199'       # git拉代码的IP地址，公网IP和内网IP均可
        gitlab_rails['gitlab_shell_ssh_port'] = 2222
        # unicorn['port'] = 8888
        # nginx['listen_port'] = 80
    ports:    #端口映射，可以自行配置
      - '9080:80'
      - '8443:443'
      - '2222:22'
    volumes:  #目录挂载，可以自定义
      - ~/Project/Docker/gitlab/config:/etc/gitlab
      - ~/Project/Docker/gitlab/data:/var/opt/gitlab
      - ~/Project/Docker/gitlab/logs:/var/log/gitlab
      - /etc/localtime:/etc/localtime:ro
    shm_size: '256m'

注解

有时候可能会因为权限问题导致无法正常创建目录/文件, 可以在文件映射后面加 Z (~/Project/Docker/gitlab/config:/etc/gitlab:Z) , 并提权 docker exec -it gitlab update-permissions (相当于运行后手动更新权限).

或者一开始就用特权 privileged: true

默认管理员账户:

#   root
#   /etc/gitlab/initial_root_password

配置使用自带nginx的https

services:
  gitlab:
    container_name: 'gitlabCe16_5_8'    #容器名
    image: 'gitlab/gitlab-ce:16.5.8-ce.0'
    restart: always
    # restart: unless-stopped
    # hostname: 'gitlab.example.com'
    environment:
      TZ: 'Asia/Shanghai'
      # | 表示下面配置是多行配置
      GITLAB_OMNIBUS_CONFIG: |
        # external_url  'http://192.168.1.108'       # git拉代码的IP地址，公网IP和内网IP均可
        # gitlab_rails['gitlab_shell_ssh_port'] = 2222
        # unicorn['port'] = 8888
        # nginx['listen_port'] = 80

        nginx['listen_addresses'] = ['127.0.0.1','*', '[::]']
        # 开启 https
        nginx['listen_port'] = 443
        external_url  'https://192.168.1.108:8443'
        nginx['redirect_http_to_https'] = true
        # 将 80 端口的转发到 external_url/url
        nginx['redirect_http_to_https_port'] = 80

    ports:   #端口映射，可以自行配置
      - '9080:80'
      - '8443:443'
      - '2222:22'
    volumes:  #目录挂载，可以自定义
      - /Volumes/WinterSamsung/Project/Docker/Gitlab/config:/etc/gitlab
      - /Volumes/WinterSamsung/Project/Docker/Gitlab/data:/var/opt/gitlab
      - /Volumes/WinterSamsung/Project/Docker/Gitlab/logs:/var/log/gitlab
      - /etc/localtime:/etc/localtime:ro
    shm_size: '256m'

主要是这一部分(及其含义)

# 配置给nginx监听地址, 我加了个127.0.0.1
nginx['listen_addresses'] = ['127.0.0.1','*', '[::]']
# 开启 https, 设置 443 端口
nginx['listen_port'] = 443

# 访问地址, 不设置默认是hostname,

# 注意:
#   默认情况下, external_url 设置了 https 开头后, 就不会再监听80端口, 除非显示设置 redirect_http_to_https

#   参考: https://gitlab.com/gitlab-org/omnibus-gitlab/blob/master/doc/settings/nginx.md#redirect-http-requests-to-https
#         https://www.coder.work/article/7371344
#   By default, when you specify an external_url starting with 'https', NGINX will no longer listen for unencrypted HTTP traffic on port 80.
#   If you want to redirect all HTTP traffic to HTTPS you can use the redirect_http_to_https setting.
external_url  'https://192.168.1.108:8443'
nginx['redirect_http_to_https'] = true
# 显示指定将 80 端口的转发到 external_url/url
nginx['redirect_http_to_https_port'] = 80

参考:

  • docker - Gitlab-ce docker容器无法通过https访问
  • 使用Docker安装配置GitLab CE
  • NGINX settings

其它-gitlab-Rest接口使用

有时候可能会有自己去实现一个客户端, 访问gitlab的一些接口, 这里记录一下.

官方文档
  • 15.11版本 https://archives.docs.gitlab.com/15.11/ee/api/api_resources.html
  • 最新版, 目前是17 https://docs.gitlab.com/ee/api/rest/

最重要的是要看自己用的那个版本的文档, 因为不同版本的接口可能会有变化. 比如15不支持下载上传的图片

ssh访问问题

使用 nginx 代理ssh转发时, 发现访问不了gitlab的ssh

nginx错误日志:

[error] 30#30: *20 connect() failed (111: Connection refused) while connecting to upstream, client: 192.168.50.76, server: 0.0.0.0:22, upstream: "172.19.0.32:22", bytes from/to client:0/0, bytes from/to upstream:0/0

其中

  • 192.168.50.76 - 访问使用的客户端
  • 192.168.50.100 - nginx地址, 也是启动的docker, 与gitlab公用一个桥接网卡
  • 172.19.0.32 - gitlab容器ip, 由 nginx 代理转发, 配置tcp转发可参考 配置 的 tcp/udp代理

查看gitlab日志发现有权限的报错:

Unable to load host key "/etc/gitlab/ssh_host_rsa_key": bad permissions

猜测应该是升级的时候, 没设置对 UID/GID , 导致迁移有问题, 先重新设置了一下容器环境

gitlabu:
image: gitlab/gitlab-ce:14.9.5-ce.0
container_name: gitlabu
hostname: gitlabu
restart: unless-stopped
environment:
  TZ: 'Asia/Shanghai'
  UID: 0
  GID: 0

# ... 省略

然后启动好去容器里手动修复权限

chmod 600 /etc/gitlab/ssh_host_*_key
chmod 644 /etc/gitlab/ssh_host_*_key.pub
chown git:git /etc/gitlab/ssh_host_*_key*

# 重新配置重启
gitlab-ctl reconfigure
gitlab-ctl restart

现在正常了

功能

允许注册

artifacts for pages are too large

参考: https://www.cnblogs.com/devilyouwei/p/14151367.html

或者是 upload artifact fails: too large

这里有两种情况

  • 一个是构建的时候报错 413, 此时是更新 CI/CD 时候的构建产物大小.
  • 另一个是构建好发布到pages的时候报错, 此时是去pages更新产物大小

开启gitlab-pages

参考
  • https://docs.gitlab.com/administration/pages/
  • [博客园] gitlab-- pages 功能实战

compose配置需要更新

# 原来的
external_url 'http://gitlab.yq.lc'
gitlab_rails['gitlab_shell_ssh_port'] = 22

# 新增
pages_external_url "http://gitlab.yq.lc/"
gitlab_pages['enable'] = true
gitlab_pages['inplace_chroot'] = true

注意新版本pages页面, 不在设置里面, 而是在部署里面了.

注解

如果是gitlab的问题, 可以多看看 /var/log/gitlab/gitlab-pages/current 文件的访问日志.

比如我遇到的访问page出现502错误, 是因为容器解析不了设置的域名, 手动加了个映射为本机就好了
