target

生成目标

比如 "target": "es2019" 生成的就是 import 形式导入的 ESM (好像不是, 后面再看看)

esModuleInterop

使用 CommonJS 代码的应用应始终启用 esModuleInterop . 原因说明: https://ts.nodejs.cn/docs/handbook/modules/appendices/esm-cjs-interop.html#使用-commonjs-代码的应用应始终启用-esmoduleinterop

TypeScript 的检查和触发是同步的，因此无错误的代码应该可以安全地在任一模式下运行。 在这种情况下禁用 esModuleInterop 的缺点是，它允许你编写语义明显违反 ECMAScript 规范的 JavaScript 代码，混淆有关命名空间导入的直觉，并使将来更难迁移到运行 ES 模块

另一方面，在由第三方转译器或打包器处理的应用中，启用 esModuleInterop 更为重要。 所有主要的打包器和转译器都使用类似 esModuleInterop 的触发策略，因此 TypeScript 需要调整其检查以匹配。（编译器总是会推断 tsc 将触发的 JavaScript 文件中会发生什么，因此即使使用另一个工具代替 tsc ，影响触发的编译器选项仍应设置为与该工具的输出尽可能匹配 可能的。）

此部分参考: 模块 - ESM/CJS 互操作性

module

参考: https://www.typescriptlang.org/docs/handbook/modules/reference.html#node16-nodenext

  • .mts/.mjs/.d.mts files are always ES modules.
  • .cts/.cjs/.d.cts files are always CommonJS modules.
  • .ts/.tsx/.js/.jsx/.d.ts files are ES modules if the nearest ancestor package.json file contains "type": "module", otherwise CommonJS modules.
