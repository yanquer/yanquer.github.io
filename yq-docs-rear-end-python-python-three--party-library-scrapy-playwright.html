scrapy-playwright是scrapy的插件, 支持使用 scrapy 时候, 动态操作网页.

安装:

pip install scrapy-playwright

settings.py配置:

# 替换默认http和/或https下载处理程序 DOWNLOAD_HANDLERS
DOWNLOAD_HANDLERS = {
    "http": "scrapy_playwright.handler.ScrapyPlaywrightDownloadHandler",
    "https": "scrapy_playwright.handler.ScrapyPlaywrightDownloadHandler",
}

# REACTOR, 需要在入口安装, 即使是使用 crawl 方式启动
#  from scrapy.utils.reactor import install_reactor
#  install_reactor('twisted.internet.asyncioreactor.AsyncioSelectorReactor')
TWISTED_REACTOR="twisted.internet.asyncioreactor.AsyncioSelectorReactor",

PLAYWRIGHT_LAUNCH_OPTIONS={
    # 是否使用无头模式, False 表示会显式打开一个浏览器窗口, 否则不会打开
    "headless": True,
    # 应该是单个窗口加载超时
    "timeout": 60 * 1000,  # 1 min
},
# 网页首次加载超时时间
PLAYWRIGHT_DEFAULT_NAVIGATION_TIMEOUT=10 * 60 * 1000,  # 10 min

spider参考:

import scrapy

class AwesomeSpider(scrapy.Spider):
    name = "awesome"

    def start_requests(self):
        # GET request
        yield scrapy.Request("https://httpbin.org/get", meta={"playwright": True})
        # POST request
        yield scrapy.FormRequest(
            url="https://httpbin.org/post",
            formdata={"foo": "bar"},
            meta={"playwright": True},
        )

    def parse(self, response, **kwargs):
        # 'response' contains the page as seen by the browser
        return {"url": response.url}

部分配置选项-Request

即:

yield scrapy.Request

使用的参数, 注意参数作为字典传递给 meta= 选项

playwright: bool = False
是否交给 Playwright 处理
playwright_context: str = "default"
传递给浏览器的上下文(类似于全局变量)
playwright_include_page: bool = False

True表示用于下载请求的 Playwright 页面, 将在回调中通过 response.meta['playwright_page'] 获取

比如

async def parse(self, response, **kwargs):
  page = response.meta['playwright_page']

  await page.wait_for_timeout(1000)
  html = await page.content()
  response = scrapy.http.HtmlResponse(url=page.url, body=html.encode(), encoding='utf-8')
  books = response.css("div.px-1 > div.grid.gap-2 > div > a")
playwright_page_methods: Iterable = ()
一个可迭代的 scrapy_playwright.page.PageMethod 对象，用于指示在返回最终响应之前要在页面上执行的操作
playwright_page: Optional[playwright.async_api._generated.Page] = None

用于下载请求的 Playwright 页面。

如果未指定，则为每个请求创建一个新页面。 该键可以与 结合使用 playwright_include_page，以使用同一页面发出一系列请求。

例如：

def start_requests(self):
  yield scrapy.Request(
      url="https://httpbin.org/get",
      meta={"playwright": True, "playwright_include_page": True},
  )

def parse(self, response):
    page = response.meta["playwright_page"]
    yield scrapy.Request(
        url="https://httpbin.org/headers",
        callback=self.parse_headers,
        meta={"playwright": True, "playwright_page": page},
    )
playwright_page_goto_kwargs: dict = {}

当导航到 URL 时，带有要传递给页面 goto方法的关键字参数的字典。

如果存在，则忽略该url键，而是使用请求的 URL。

playwright_security_details: Optional[dict]

只读

包含有关给定响应的安全信息的字典。 仅适用于 HTTPS 请求。可以通过回调在回调中访问 response.meta['playwright_security_details']

部分配置选项-settings.py

PLAYWRIGHT_ABORT_REQUEST: Optional[Union[Callable, str]] = None

不处理的资源类型

它接收一个 playwright.async_api.Request对象并且True如果请求应该被中止则必须返回，False否则。 支持协程函数 ( async def)。

例如，以下都是等效的，并阻止下载图像

PLAYWRIGHT_ABORT_REQUEST = lambda req: req.resource_type == "image"

def should_abort_request(req):
    return req.resource_type == "image"

PLAYWRIGHT_ABORT_REQUEST = should_abort_request

# project/utils.py
def should_abort_request(req):
    return req.resource_type == "image"

# settings.py
PLAYWRIGHT_ABORT_REQUEST = "project.utils.should_abort_request"

示例-在回调中接收 Page 对象

注意需要将回调定义为协程函数 async def

警告

请谨慎使用，并且仅当您确实需要在回调中对 Page 对象执行操作时。

如果页面在不再需要后没有正确关闭，则蜘蛛作业可能会因为设置 PLAYWRIGHT_MAX_PAGES_PER_CONTEXT 设置的限制而卡住

import scrapy

class AwesomeSpiderWithPage(scrapy.Spider):
    name = "page_spider"

    def start_requests(self):
        yield scrapy.Request(
            url="https://example.org",
            callback=self.parse_first,
            meta={"playwright": True, "playwright_include_page": True},
            errback=self.errback_close_page,
        )

    def parse_first(self, response):
        page = response.meta["playwright_page"]
        return scrapy.Request(
            url="https://example.com",
            callback=self.parse_second,
            meta={"playwright": True, "playwright_include_page": True, "playwright_page": page},
            errback=self.errback_close_page,
        )

    async def parse_second(self, response):
        page = response.meta["playwright_page"]
        title = await page.title()  # "Example Domain"
        await page.close()
        return {"title": title}

    async def errback_close_page(self, failure):
        page = failure.request.meta["playwright_page"]
        await page.close()
  • 传递时 playwright_include_page=True，请确保页面不再使用时始终关闭。 建议设置 Request errback 以确保即使请求失败也关闭页面（如果 playwright_include_page=False 或未设置，页面会在遇到异常时自动关闭）。 这很重要，因为打开的页面计入设置的限制， PLAYWRIGHT_MAX_PAGES_PER_CONTEXT 如果达到限制并且页面无限期保持打开状态，爬网可能会冻结。
  • async def 仅在需要时才需要定义回调， await 如果您只需要将 Page 对象从一个回调传递到另一个回调（参见上面的示例），则没有必要。
  • goto 任何因在 Page 对象（ ,等）上等待协程而产生的网络操作 go_back 都将由 Playwright 直接执行， 绕过 Scrapy 请求工作流程（调度程序、中间件等）。

问题-共用page时多个页面的阻塞

当使用参数选项 meta={"playwright": True, "playwright_include_page": True, "playwright_page": page},

如果全局都是共享一个 page, 问题貌似不大.

如果是在 def start_requests 中不共享:

meta={"playwright": True, "playwright_include_page": True}

async def parse 中才开始共享:

meta={"playwright": True, "playwright_include_page": True, "playwright_page": page}

可能会出现阻塞问题.

为什么?

  • 首先, playwright打开浏览器页面的操作是互斥的.

考虑设置并发数为1的情况

  • def start_requests 返回了 5 个 yield

    def start_requests(self):
  for i in range(5):
    yield scrapy.Request(
        url="https://example.org",
        callback=self.parse_first,
        meta={"playwright": True, "playwright_include_page": True},
        errback=self.errback_close_page,
    )

  • 由于并发数为1, 首先执行第一个 scrapy.Request, 称之为任务1

  • 在 async def parse 共享page, 需要查询的url是多页

    async def parse(self):
  page = response.meta["playwright_page"]
  last_page = False
  # .....

  # yield item

  if last_page:
    await page.close()
  else:
    yield scrapy.Request(
        url="https://example.org",
        callback=self.parse_first,
        meta={"playwright": True, "playwright_include_page": True, "playwright_page": page},
        errback=self.errback_close_page,
    )

  # ......

  • 由于共享了 page, 第一页找完数据后, 会将page传递给 yield scrapy.Request, 同时, 浏览器页面不会关闭

  • 因为浏览器页面不会关闭, 同时前面也提过 playwright 是阻塞的, 所以 def start_requests 的第 2 个 yield, 就会一直等待浏览器页面关闭后再执行.

注解

这里先忽略掉异步编程中优先级问题, 因为在 scrapy 的异步编程中, 往往是先执行 def start_requests 返回的 yield.

另外, yield 的内容是基本按照顺序来调用的, 与 async def 定义的内容是放在事件循环中执行的顺序不同.

解决方案
  • 保证你的任务并发数比 CONCURRENT_REQUESTS 设置额的比 def start_requests 返回的 yield 数目大
