将应用构建成包

比较流行的使用方式是在项目目录下放置一个 electron-builder.yml 文件

# id标识
appId: com.yq.flower

# 产品名称, 一般用作打包名称
productName: Screen Flower

# 版权
copyright: Copyright © 2024 YanQue

# 目录配置
directories:
  # 输出位置
  output: dist
  buildResources: resources

# 打包文件配置
files:
  - from: .
    filter:
      - package.json
      - app
  # 排除 .vscode 目录下的所有文件
  - '!**/.vscode/*'
  # 排除 src 目录下的所有文件
  - '!src/*'
  # 排除配置文件
  - '!electron.vite.config.{js,ts,mjs,cjs}'
  - '!{.eslintignore,.eslintrc.cjs,.prettierignore,.prettierrc.yaml,dev-app-update.yml,CHANGELOG.md,README.rst}'
  # 排除环境变量文件和 npm 配置文件
  - '!{.env,.env.*,.npmrc,pnpm-lock.yaml}'

# 发布设置
publish: null

# 是否使用 asar 压缩
#asar: false

# 额外配置打包的资源文件
extraResources:
#  - resources
  - from: resources
    to: resources

## 指定从asar归档中解压的文件或文件夹
#asarUnpack:
#  # 解压 resources 目录下的所有内容
#  - resources/**

# 不需要签名公证
#afterSign: "electron-builder-notarize"
# mac下打包dmg配置
dmg: {
  icon: 'resources/icons/screen-flower.icns',
  background: 'resources/icons/bg.jpg',
  contents: [
    { # 这个是右边图标及内容
      x: 410,
      y: 190,
      type: 'link',
      path: '/Applications'
    },
    { # 你左边的图标
      x: 130,
      y: 190,
      type: 'file'
    }
  ],
  window: { # 这里是整个窗口的大小
    height: 380,
    width: 540
  }
}

mac: {
  # 应用程序图标
  icon: 'resources/icons/screen-flower.icns',
  # 应用程序包名
  artifactName: '${productName}-${platform}-${arch}-${version}.${ext}',
  target: [ # 要打的包的格式类型设置
            'dmg',
            'zip' # 这里注意更新的时候，mac只认zip格式的包
  ],
  # 是否进行苹果官方的签名
  notarize: false,

  # 安装后执行文件名称, 默认是 package.json 中的 name 字段 (windows下默认是 productName)
  # executableName: xxx
}

注解

electron-builder 会智能的把一些 native 的程序（包括 exe执行文件 ）打包到 app.asar.unpacked 中

关于mac配置项

notarize: boolean
是否需公证签名
hardenedRuntime: boolean

启用 macOS 的 Hardened Runtime 机制，强制应用在沙箱中运行，限制危险操作（如动态代码注入、未签名的代码执行等）。

必须启用才能通过公证（Notarization），否则应用会被 macOS Gatekeeper 拦截。

gatekeeperAssess: boolean

控制构建时是否触发 codesign 的 --deep 签名验证和 Gatekeeper 兼容性检查。

设置为 false 时，跳过构建阶段的 Gatekeeper 检查，可能加快构建速度。 不影响最终用户运行时的 Gatekeeper 行为，用户首次打开应用时仍会触发 Gatekeeper 检查。

entitlements: string
授权文件（.entitlements），声明应用所需的系统权限（如访问摄像头、网络、文件系统等）。 主进程（Main Process）的权限声明。
entitlementsInherit: string
授权文件（.entitlements），声明应用所需的系统权限（如访问摄像头、网络、文件系统等）。 子进程（如渲染进程、子进程）继承的权限。

关于macOS签名

签名分两步, 一个是对本地代码进行签名, 一个是用AppleId公证签名(以便发布到应用商店).

对于没有去Apple申请开发者证书的情况来说, 我们只进行本地代码签名即可

本地代码进行签名

不进行本地签名的坏处

  • 如果你将打包好的包安装的本机 /Applications/app.app, 你的应用需要获取录屏权限, 然后你手动在系统设置里将录屏权限赋予给 /Applications/app.app.

    你以为这就结束了? 不, 当进行实际需要录屏操作时候, /Applications/app.app/Contents/macOS/app 还会申请一次权限.

    为什么对 /Applications/app.app 的权限无法附给 /Applications/app.app/Contents/macOS/app. 这就是没有进行代码签名的问题了. 如果进行了代码签名, 两者就是一体(这么理解)

如果进行签名? 笔者没有去申请Apple开发者证书, 所以使用本机自动生成的本地开发证书. 可以在这里导出

注解

如果没有, 用自己的AppleId登陆即可, 会生成一个仅本地可用的证书. 如果上次生成的过期了, 重新登陆还会再重新生成一个.

导出时候配置一下导出路径与密码(即设置给访问证书的密码), 比如导出到 ~/project/cert/dev.p12, 密码是 12345. 那么需要去环境变量中配置:

export CSC_LINK=/Users/用户名/project/cert/dev.p12
export CSC_KEY_PASSWORD=12345
# 可选
# export CSC_NAME

笔者是在默认的 ~/zshrc 中增加的.

接着直接默认配置执行 electron-builder 即可

注解

如果你不想进行代码签名, 设置环境变量 CSC_IDENTITY_AUTO_DISCOVERY=false 即可

export CSC_IDENTITY_AUTO_DISCOVERY=false && electron-builder --config electron-builder.yml

注解

如果你的证书过期了, 打开APP会出现 恶意应用 的提示, 需要重新导出有效的证书. 如果你使用的还是过期的证书, 代码签名的时候就会报错 app: CSSMERR_TP_CERT_REVOKED

AppleId公证签名

一般是使用 afterSign: "electron-builder-notarize"`

问题

macOS打包的asar内的执行文件签名

可能会遇到这样的报错信息:

signature in <...> not valid for use in process: mapped file has no chash, completely unsigned? Code has to be at least ad-hoc signed.

即没有自动签名node_modules中的二进制文件.

当启用了 asar 打包时候, 某些 .node 执行文件被打包进了 asar 包内. 笔者出现问题的包是 node-mac-permissions , 最后是在 electron-builder.yml 加了个这配置:

"asarUnpack": [
  # 解决放到asar会出现模块未签名的问题
  "node_modules/node-mac-permissions"
]

也可以直接:

"build": {
  "mac": {
    "asarUnpack": "node_modules/**/*.node"
  }
}

注解

还可以不设置 entitlements, 或者在 entitlements 的plist中配置:

<key>com.apple.security.cs.disable-library-validation</key>
<true/>

但是不建议, 因为这样就是mac的安全机制冲突, 高版本运行还是会有问题

配置项说明 (AI生成)

menuCategory

控制是否在 Windows 的“开始菜单”中为应用程序创建一个分类文件夹。

  • true，安装程序会在“开始菜单”中创建一个以应用程序名称命名的文件夹，并将快捷方式放入其中。
  • false，快捷方式会直接放在“开始菜单”的根目录中。

在 NSIS 脚本中，这通常通过 CreateShortCut 和 SetShellVarContext 实现。

CreateDirectory "$SMPROGRAMS\YourAppName"
CreateShortCut "$SMPROGRAMS\YourAppName\YourApp.lnk" "$INSTDIR\YourApp.exe"
oneClick

是否启用“一键安装”模式。

  • true，安装程序会跳过所有用户交互（如选择安装路径、确认安装等），直接安装到默认路径。
  • false，安装程序会显示标准的安装向导，允许用户选择安装路径和其他选项。

当 oneClick 为 false 时，NSIS 脚本会包含以下内容：

; 启用标准的安装向导页面。
!define MUI_PAGE_CUSTOMFUNCTION_SHOW ShowInstFiles
!insertmacro MUI_PAGE_INSTFILES
perMachine

控制安装程序是为当前用户安装还是为所有用户安装。

  • true，安装程序会将应用程序安装到系统目录（如 C:Program Files ），并为所有用户创建快捷方式。
  • false，安装程序会将应用程序安装到当前用户的目录（如 C:UsersYourUsernameAppDataLocal ），并仅为当前用户创建快捷方式。

当 perMachine 为 true 时， NSIS 脚本会设置安装目录为系统目录：

InstallDir "$PROGRAMFILES\YourAppName"

当 perMachine 为 false 时， NSIS 脚本会设置安装目录为用户目录：

InstallDir "$LOCALAPPDATA\YourAppName"

winCodeSign执行失败

不管是下载失败, 还是下载好后解压失败, 这玩意儿不要想着去禁用签名 sign: false , 没用

默认下载会把文件下载到 C:Users用户名AppDataLocalelectron-builderCachewinCodeSign, 如果是交给工具下载, 会先下载一个一串数字编码的7z包, 比如 C:Users用户名AppDataLocalelectron-builderCachewinCodeSign702973027.7z, 然后自动解压.

我是虚拟机解压的时候报错格式问题, 无奈只好手动将 702973027.7z 解压到 C:Users用户名AppDataLocalelectron-builderCachewinCodeSignwinCodeSign-2.6.0, 注意解压包的版本号
